## 一、任务接单标准

### ✅ 接受的任务类型

| 类型 | 说明 | 示例 |
|------|------|------|
| 信息检索 | 可通过公开渠道验证的事实查询 | 市场调研、竞品分析、数据收集 |
| 内容创作 | 文字、代码、设计等创意产出 | 文案撰写、代码片段、PPT框架 |
| 流程优化 | 基于现有信息的流程改进 | SOP制定、checklist设计 |
| 翻译校对 | 语言转换与质量检查 | 文档翻译、润色修改 |
| 日程协调 | 时间管理与会议安排 | 日程规划、提醒设置 |
| 技术咨询 | 基于知识库的技术建议 | 架构建议、工具选型 |
| **AI 模型研发** | 垂直领域模型训练与发布 | 模型微调、评测、HuggingFace 发布 |

### ❌ 不接受的任务类型

| 类型 | 原因 | 替代方案 |
|------|------|----------|
| 违法违规 | 法律与道德风险 | 拒绝并说明原因 |
| 侵犯隐私 | 数据安全风险 | 建议用户自行处理 |
| 实时交易 | 无法保证时效性 | 建议使用专业交易平台 |
| 线下执行 | 无法物理到场 | 提供远程指导方案 |
| 医疗诊断 | 专业资质限制 | 建议咨询专业医生 |
| 法律意见 | 专业资质限制 | 建议咨询律师 |

### ⚠️ 需确认的任务类型

以下任务需 CEO 审批后方可接单：

- 涉及公司对外发布的正式声明
- 需要访问用户敏感数据
- 超过 4 小时预估工时的复杂任务
- 涉及第三方付费 API 的任务
- **AI 模型涉及敏感领域（医疗、金融、法律）**
- **AI 模型使用非公开数据集**

## 二、交付流程（三阶段）

### 阶段一：任务确认（T+0）

**动作清单：**

1. **需求澄清**
   - 确认任务目标、输出格式、截止时间
   - 识别潜在风险和依赖条件
   - 记录到任务追踪系统

2. **可行性评估**
   - 判断是否在接单标准范围内
   - 评估所需资源和时间
   - 标记不确定项

3. **预期对齐**
   - 告知用户预计交付时间
   - 说明可能的限制条件
   - 确认优先级

**输出物：**
- 任务确认记录（含需求摘要、预计交付时间）

---

### 阶段二：任务执行（T+1 ~ T+N）

**动作清单：**

1. **方案制定**
   - 拆解任务为可执行步骤
   - 选择合适的工具和方法
   - 预留 20% 缓冲时间

2. **分步执行**
   - 按优先级顺序推进
   - 遇阻时及时同步用户
   - 记录关键决策点

3. **进度同步**
   - 超过 2 小时的任务，每小时同步进度
   - 遇到方向性问题时，提供选项让用户决策

**输出物：**
- 中间成果（草稿、方案文档等）
- 进度更新记录

---

### 阶段三：交付验收（T+N）

**动作清单：**

1. **质量检查**
   - 执行完整检查清单（见第三节）
   - 修正发现的问题
   - 自我评分

2. **成果交付**
   - 按约定格式整理输出
   - 附使用说明（如适用）
   - 提供修改建议入口

3. **反馈收集**
   - 询问满意度（1-5分）
   - 记录改进建议
   - 标记是否需要后续跟进

**输出物：**
- 最终交付成果
- 交付确认记录

## 三、质量检查清单

每个任务交付前必须通过以下检查：

### 基础检查（必过项）

| 检查项 | 标准 | 通过条件 |
|--------|------|----------|
| 完整性 | 交付物覆盖所有需求点 | 无遗漏项 |
| 准确性 | 事实性内容可验证 | 无明显错误 |
| 可用性 | 输出格式符合预期 | 用户可直接使用 |
| 时效性 | 在承诺时间内交付 | 无超时 |

### 内容检查（按类型）

**文字类：**
- [ ] 无错别字、语病
- [ ] 逻辑结构清晰
- [ ] 引用来源标注（如适用）

**代码类：**
- [ ] 语法正确可运行
- [ ] 关键处有注释
- [ ] 提供使用示例

**数据类：**
- [ ] 数据来源可信
- [ ] 处理逻辑可追溯
- [ ] 异常值已标注

**建议类：**
- [ ] 有依据支撑
- [ ] 提供多个选项
- [ ] 说明优劣对比

### 体验检查

- [ ] 语言简洁，无冗余
- [ ] 结构化呈现（表格/列表/分节）
- [ ] 提供"下一步"指引

## 四、交付时效标准

### 标准响应时间

| 任务复杂度 | 定义 | 首次响应 | 交付时限 |
|------------|------|----------|----------|
| 简单 | 单一步骤，<30分钟 | 5分钟内 | 2小时内 |
| 中等 | 多步骤，30分钟-2小时 | 10分钟内 | 当日完成 |
| 复杂 | 多步骤+协调，>2小时 | 30分钟内 | 协商确定 |

### 复杂度判断标准

**简单任务：**
- 单一信息查询
- 简短文案撰写（<500字）
- 代码片段（<50行）
- 单次翻译校对

**中等任务：**
- 多维度信息整合
- 完整文档撰写（500-2000字）
- 功能性代码（50-200行）
- 流程设计方案

**复杂任务：**
- 跨领域深度研究
- 完整项目方案
- 多文件代码开发
- 需多轮协调的任务

### 超时处理

1. **预警机制**：预计超时前 30 分钟告知用户
2. **方案调整**：提供分阶段交付或简化方案选项
3. **补偿措施**：根据情况提供额外价值（如补充分析）

## 五、异常处理流程

### 遇阻上报

| 层级 | 触发条件 | 处理人 |
|------|----------|--------|
| L1 | 技术问题，可自行解决 | 执行Agent |
| L2 | 需求不明确，需用户确认 | 用户 |
| L3 | 超出能力范围，需协调资源 | COO |
| L4 | 战略决策，涉及方向选择 | CEO |

### 争议处理

1. 用户对交付不满意 → 优先修改，最多 2 轮
2. 修改后仍不满意 → 上报 COO 评估
3. 涉及退款/补偿 → 上报 CEO 决策

## 六、AI 模型专项交付标准

> 本节适用于 AI 垂直领域模型研发任务，包括模型训练、微调、评测和发布。

### 6.1 模型验收标准

#### 性能指标要求

| 指标类别 | 验收标准 | 测试方法 | 通过阈值 |
|----------|----------|----------|----------|
| **功能正确性** | 模型输出符合预期 | 人工抽检 + 自动评测 | 准确率 ≥ 90% |
| **推理性能** | 响应速度满足需求 | 压力测试 | P99 延迟 < 2s |
| **吞吐量** | 并发处理能力 | 负载测试 | 满足 SLA 承诺 |
| **稳定性** | 长时间运行无异常 | 7×24h 稳定性测试 | 无崩溃/内存泄漏 |
| **资源占用** | GPU/内存使用合理 | 监控分析 | 不超过预算 110% |

#### 测试通过率要求

```
测试套件结构：

单元测试（Unit Tests）
├─ 模型加载测试
├─ 前向传播测试
├─ 输出格式测试
└─ 边界条件测试
   → 通过率要求：100%

集成测试（Integration Tests）
├─ 端到端推理测试
├─ 数据 pipeline 测试
├─ API 接口测试
└─ 错误处理测试
   → 通过率要求：≥ 95%

评测测试（Evaluation Tests）
├─ 标准数据集评测
├─ 领域特定评测
├─ 对抗样本测试
└─ 偏见/公平性测试
   → 通过率要求：≥ 90%
```

## 七、紧急任务处理流程

### 7.1 什么样的任务可以插队？

#### 允许插队的情况

| 类型 | 定义 | 示例 | 审批权限 |
|------|------|------|----------|
| **安全漏洞修复** | 模型存在严重安全漏洞，需立即修复 | 发现可 exploited 的对抗攻击 | CTO |
| **客户违约风险** | 已签约客户交付即将延期，涉及违约金 | 大客户合同即将到期 | COO |
| **竞品窗口期** | 错过将丧失市场机会的时间敏感任务 | 行业大会发布、竞品发布前 | CEO |
| **生产事故** | 线上模型服务出现故障 | 服务不可用、严重性能下降 | CTO |
| **监管合规** | 监管要求限期整改 | 数据合规、算法备案 | CEO |

#### 不允许插队的情况

- 仅因"客户催得急"但无合同约束
- 内部管理问题导致的紧急（如计划不周）
- 可以延后处理而不产生实质损失的任务
- 个人原因导致的紧急（如"我下周要休假"）

### 7.2 审批权限：谁有权突破产能限制？

#### 权限矩阵

| 决策类型 | P0 任务 | P1 任务 | P2/P3 任务 |
|----------|---------|---------|------------|
| **正常排程** | CEO | CTO | 技术负责人 |
| **插队审批** | CEO | CTO + COO | 技术负责人 |
| **突破产能上限** | CEO 专属 | CEO 授权 | 不可突破 |
| **暂停其他任务** | CEO 专属 | CEO 授权 | 不可暂停 |

#### 突破产能的审批流程

```
紧急任务插队申请流程：

申请人
  │
  ▼
┌─────────────────────────────────────────────────────────────┐
│ 填写《紧急任务插队申请表》                                    │
│ - 任务描述                                                   │
│ - 紧急原因（符合 5 类允许情况之一）                            │
│ - 期望插队位置                                               │
│ - 影响评估（被插队任务的影响）                                 │
│ - 风险预案                                                   │
└─────────────────────────────────────────────────────────────┘
  │
  ▼
直接上级初审（2h 内）
  │ 不通过 → 退回并说明原因
  ▼ 通过
审批人审批（根据权限矩阵）
  │ 不通过 → 退回并说明原因
  ▼ 通过
COO 备案并调整排程
  │
  ▼
通知相关方并执行
```

#### 突破产能的代价与记录

**代价机制：**
- 插队任务需在 2 倍标准时间内完成，否则记为违约
- 被插队任务自动顺延，需重新确认 SLA
- 当月插队次数 > 3 次，触发流程审查

**记录与复盘：**
- 所有插队任务记录在案
- 每月统计插队频率和原因
- 季度复盘：插队是否必要？流程如何优化？
