## 一、当前技术栈分析

### 现有架构

| 层级 | 技术选型 | 状态 | 备注 |
|------|---------|------|------|
| 前端框架 | React 18 + TypeScript | ✅ 已上线 | 成熟稳定 |
| 构建工具 | Vite 5 | ✅ 已上线 | 快速 HMR，适合迭代 |
| 样式方案 | 纯 CSS（index.css） | ⚠️ 待优化 | 无 CSS 框架，缺少设计系统 |
| 后端 | 无 | ❌ 缺失 | 落地页纯静态，无法收集用户数据 |
| 数据库 | 无 | ❌ 缺失 | 无法存储 waitlist 数据 |
| 部署 | 未知 | ⚠️ 待确认 | dist/ 已构建，需确定 CDN/主机 |

### 关键判断

- **前端基础扎实**：React + TS + Vite 是当前最佳前端组合之一，技术债低。
- **核心缺口在后端**：无 API 层、无数据库、无用户体系——这是 MVP 必须填补的能力。
- **样式体系薄弱**：缺少设计 tokens、组件库，不利于后续快速迭代。
- **部署链路不明**：需立即确认 CI/CD 和托管方案。

## 二、技术投入与里程碑

### 阶段一：短期（1-3 个月）—— MVP 核心功能

**目标**：从落地页进化为可运营的产品，支撑种子用户验证。

#### 1.1 后端基础设施搭建（Week 1-3）

```
技术选型：
  - 运行时：Node.js 20 LTS + Express / 备选：Hono（更轻量）
  - 数据库：PostgreSQL（Supabase / Neon）
  - ORM：Prisma
  - 部署：Vercel Functions / Railway / Fly.io
  - API 规范：REST → 快速验证；GraphQL 暂不考虑

技术投入估算：
  - 1-2 名后端工程师
  - 预计 3-4 周完成基础 CRUD API 层

风险点：
  ⚠️ 数据库选型分歧：PostgreSQL vs 轻量替代（Planetscale/Neon）
  → 建议：PostgreSQL + Supabase，成熟度高，有免费额度
  ⚠️ 无 DevOps 经验：建议直接用 Supabase Managed PostgreSQL，减少运维负担
```

#### 1.2 Waitlist 功能上线（Week 2-4）

→ 详见 `WAITLIST_TECH.md`

#### 1.3 用户认证体系（Week 3-5）

```
技术选型：
  - 方案：Supabase Auth / Clerk / 自建
  - 推荐：Supabase Auth（与 DB 同体系，整合成本最低）

核心功能：
  - 邮箱 + 密码注册登录
  - Magic Link（无密码登录，用户体验更佳）
  - OAuth（Google/GitHub 登录，可延后）

技术投入：约 1 周
风险点：⚠️ 首次接入 Auth 是安全漏洞高发区，建议直接用成熟方案，不自建
```

#### 1.4 种子功能开发（Week 4-8）

> 根据产品定位，本阶段应实现 杭州市上城区乐友信息服务工作室 的核心价值主张——"One service. Any task."

```
建议：
  - 优先实现一个"杀手级"单一功能，而非多个半成品
  - 推荐 AI Agent 任务执行入口（杭州市上城区乐友信息服务工作室 的核心差异化）
  - 前端状态管理：Zustand（轻量）或 TanStack Query（推荐，数据获取+缓存）

技术栈补充：
  - AI 集成：OpenAI API / Claude API / 国产模型（待定）
  - AI 路由：LiteLLM（统一调用接口，便于切换模型）
```

#### 1.5 样式体系建立（Week 1-8 并行）

```
技术选型：
  - Tailwind CSS（推荐：开发效率高，原子化，与设计系统天然契合）
  - 设计 Tokens：使用 CSS Variables 统一管理颜色、间距、字体
  - 组件库：Radix UI（无样式 Headless 组件）+ Tailwind，替代笨重的 MUI/Antd

并行任务，不阻塞功能开发
```

### 阶段二：中期（3-6 个月）—— 平台化扩展

**目标**：从单一功能产品进化为可扩展平台，支撑增长。

#### 2.1 支付体系

```
技术选型：
  - Stripe（国际）
  - 微信/支付宝（国内，如主攻国内市场）
  - 推荐：先上 Stripe，再按需扩展国内支付

核心功能：
  - 免费/付费订阅分层
  - 一次性购买（按任务/按月）
  - 计费系统：基于 AI Token 消耗精确计费
```

#### 2.2 开放 API / Webhooks

```
目的：
  - 让用户和开发者将 杭州市上城区乐友信息服务工作室 集成到自己的工作流
  - 生态建设的第一步

技术方案：
  - REST API v1，带版本控制（/v1/）
  - API Key 鉴权（Supabase/Clerk 支持）
  - Webhooks：接收事件推送（如任务完成、订阅变更）
```

#### 2.3 平台化架构演进

```
问题：单体架构在功能增加后会变成大泥球（Big Ball of Mud）

解决方案：
  - 引入微前端（如 qiankun）隔离大型功能模块
  - 服务端：Node.js → NestJS（如果业务复杂度上升）
  - 引入消息队列（Redis/BullMQ）：异步处理 AI 任务，不阻塞请求
  - 引入缓存层：Redis（会话缓存、限流、Rate Limiting）
```

#### 2.4 监控与可观测性

```
必选（Growth Stage）：
  - 错误监控：Sentry
  - 访问分析：Plausible / Posthog（隐私友好）
  - 日志：结构化日志（pino）+ 日志聚合（Logflare/Datadog）
```

### 阶段三：长期（6-12 个月）—— 规模化基础设施

**目标**：支撑数百万用户、企业级可靠性。

#### 3.1 多租户与权限体系

```
需求：
  - 企业版：团队协作、权限控制、审计日志
  - 租户隔离：数据安全是生死线

技术方案：
  - 数据库行级安全（RLS）配合 Supabase
  - RBAC 权限系统：Casbin 或自定义
```

#### 3.2 全球加速与边缘计算

```
技术选型：
  - CDN：Cloudflare（边缘节点覆盖广，免费额度充足）
  - 边缘函数：Cloudflare Workers（低延迟 AI 任务路由）
  - 数据库多区域：Supabase Multi-region / PlanetScale
```

#### 3.3 安全加固

```
必做：
  - WAF（Web Application Firewall）：Cloudflare Pro
  - DDoS 防护：同上
  - 安全审计：自动化依赖漏洞扫描（Dependabot + Snyk）
  - 渗透测试：上线前至少一次第三方审计
```

#### 3.4 容灾与数据备份

```
- 数据库：自动每日快照 + 跨区域备份
- 关键操作：事件溯源（Event Sourcing）记录所有状态变更
- 恢复演练：每季度一次 RTO/RPO 演练
```

#### 3.5 性能与成本优化

```
前端：
  - Lighthouse CI 集成到 PR 检查，Score ≥ 90
  - 资源优化：图片 CDN（Cloudflare Images）、字体子集化

后端：
  - 数据库查询优化（Prisma + 索引优化）
  - AI Token 缓存与复用（降低 API 成本的关键）
  - K8s / 容器化（当用户量 > 10 万时）
```

## 三、技术债务优先级矩阵

| 优先级 | 债务项 | 影响 | 修复成本 | 建议 |
|--------|--------|------|----------|------|
| P0 | 无后端 API 层 | 阻塞所有后端功能 | 高 | 立即开始 |
| P0 | Waitlist 数据存储 | 用户流失，无法运营 | 低 | 1 周内上线 |
| P1 | 样式体系混乱 | 开发效率低，品牌不一致 | 中 | Week 1-4 并行 |
| P1 | 无错误监控 | 线上问题无感知 | 低 | Week 3 上线 |
| P2 | 无 CI/CD | 部署靠手动，易出错 | 中 | Week 4 |
| P2 | 无 Analytics | 产品决策盲飞 | 低 | Week 2 |

## 四、关键决策建议（CTO 直言）

1. **不要自建 Auth**：直接用 Supabase Auth 或 Clerk，省下 2 周时间做核心功能。
2. **先支付后平台**：Payment 是规模化收入的基础，Stripe 接入是中期第一个大里程碑。
3. **AI 模型不要硬绑定**：用 LiteLLM 做抽象层，随时切换模型供应商，避免被 OpenAI 定价绑架。
4. **技术债要主动偿还**：代码审查（Code Review）是技术债的入口，每天 15 分钟 PR Review，回报率极高。
5. **安全不是后期问题**：GDPR/CCPA 合规、数据加密从一开始就要考虑，企业客户不会用不安全的产品。
